Have you ever try to scrape a website but receive no data from its result? Websites are increasingly dynamic and rendered on the client side, and more often than not, scraping HTML DOM results in empty responses since data are loaded after the HTML is received.

The first step before embarking in big data research, especially those on web platforms and applications, is collecting the data. Scraping has become a common method of data collection, and there are many ways to scrape available. One of the most common and often being taught is scraping by traversing through HTML DOM using automated functions such as Chrome extensions or scripting headless browsers through puppeteer or nightmare.

There is one other way that is not often discussed and more of a novelty, and that is to scrape by making direct HTTP requests to HTTP endpoints that power the application or website. This technique is in fact a common practice in web development, and I see the potential of repurposing this technique for data scraping. Also, this happens to be my favorite way of scraping.

Before we get into the scraping method, it is worth discussing how websites and applications are often built today and the difference between server side and client side rendering.

Terminologies

Here are some terminologies used throughout the article to get up to speed for those who are not too familiar with web development.

  1. Server Side Rendering — is the ability of an application to contribute by displaying the web-page on the server instead of rendering it in the browser.
  2. Client Side Rendering — a technique for rendering content in the browser using JavaScript.
  3. Single Page Application — is a web application or website that interacts with the web browser by dynamically rewriting the current web page with new data from the web server, instead of the default method of the browser loading entire new pages.
  4. HTTP endpoints — endpoint is a connection point where data, HTML files, or active server pages are exposed.
  5. HTML DOM (Document Object Model) — The Document Object Model (DOM) is a programming API for HTML and XML documents. It defines the logical structure of documents and the way a document is accessed and manipulated.
  6. JSON (Javascript Object Notation) — JSON (JavaScript Object Notation) is a lightweight data-interchange format. It is easy for humans to read and write. It is easy for machines to parse and generate.

Server Side vs Client Side Rendering

Websites are no longer as simple as they were. Many took the form and functionalities of applications, giving users a richer and vibrant experience. Along with this evolution in website creation, there are many tools and frameworks such as React.js and Vue.js, that fundamentally changed how websites are created. Popular services such as Airbnb, Instagram, and Facebook are built on top of these paradigms. Along with these changes, the way we scrape and access data may have to change as well.

Server side rendering is how websites are commonly served. When you open a URL link, your browser creates a request to the server requesting the website. The server responds by giving back the full HTML markup of the website, which later appears on the browser. Most scraping methods are designed for server side rendered websites in mind. For example, using Chrome extensions or headless browsers encourage you to identify HTML elements and traverse through them to collect data.

Client side rendering is an increasingly popular way to serve websites, especially those that have dynamic content. Single Page Applications (SPA) refers to websites that utilize client side rendering to serve dynamic content. Through SPA and client side rendering, when our browsers access a URL link, what the server returns and we initially receive are an empty shell HTML file with no content. However, a Javascript file is included in the HTML that will hydrate and populate the empty HTML with content. This process of hydrating and populating is done by making additional access to the server for data after the page load.

That is why scraping data from SPA and client side rendered websites is challenging since most techniques will provide users with empty data. We see the website with its content as a whole since our browser does the work on making additional requests and populating the data. However, our scraping tool will most likely only see the empty HTML shell and not be able to make the additional request as our browser does. Even if we tune our scraper to make additional requests, the scraping process can become slow. This is also why SPA and client side rendered websites have SEO issues since crawlers also struggle in loading these sites well.

Direct HTTP Request

Given the behavior of client side rendered websites, the approach for direct HTTP request becomes interesting. What if we know the endpoints that the website use to request data and make those requests directly? This approach taps into the process where client side rendered sites make additional requests to its server.

This approach is interesting and has benefits such as:

  1. Scraping becomes faster since we make a direct HTTP request and never need to load the full web page.
  2. Data that we receive is richer since servers often provide more data than what the website shows.
  3. It is as simple as making an HTTP request. In fact, after identifying and obtaining the link, we can open that link in a new tab and see the results (often in JSON form) in its bare form.
  4. Suitable for client side rendered websites, SPA or websites that incorporate infinite loading, especially those that heavily rely on REST API or GraphQL
  5. We obtain structured data from the HTTP response that we can instantly download to file. This saves a lot of time compared to matching and mapping selectors to each HTML DOM (I personally hated this process).

It has its caveat tough:

  1. We must make sure that the websites we are trying to scrape in fact use client side rendering method of populating data.
  2. Identifying the HTTP endpoint requires some trial and error and often it’s not a universal solution since each website will have different implementations.
  3. Require some workaround in making an HTTP request to go over protection measures such as Cross-Origin Resource Sharing, Same-Origin policies, Cookies, and Authentication Tokens.
  4. When the website updates its endpoint, we will also need to update our scraper algorithm and accessible endpoints.

Hence, we will be making these HTTP requests as if we are the client/website themselves, the same way we do with our own applications to our APIs. This method is actually easier than it sounds.

Scraping Airbnb using Direct HTTP Request

Now that we have an understanding of this method and its prerequisites, let’s try applying this on Airbnb and see what data we can obtain.

The objective of this exercise is to scrape Airbnb listings data from a location based on our search query in Airbnb.

The first step is to study and identify HTTP endpoint. So let us first open Airbnb.com

Next, we will search for a location on Airbnb. For this exercise, the check-in and check-out date can be random and we will not bother with the filters to keep things simple.

Now that we have the Airbnb page with our preferred location loaded, let us open our browsers’ Developer Tools. Developer Tools is a set of tools available on your browser to debug and inspect websites. This article will be based on Chrome Developer Tools, but a similar process applies to Firefox and other browsers you prefer.

On Chrome Developer Tools, we will want to navigate to the Network tab. The Network tab can record all incoming and outgoing network connections. This is where we will identify our HTTP endpoint. As you can see, after the page load, there are many additional outgoing and incoming requests made by the site. If you don’t see anything, you may need to refresh your page after opening the Developer Tools Network tab. One of these requests contains the datasets for Airbnb listings.

An easy way to find the endpoint is by using the search function available in the Network tools. Simply, try to search for the title of an Airbnb property. An endpoint containing data of the listings will appear; these data are often returned in JSON format.

By clicking on the result, we can study how the request to the HTTP endpoint is done. We can see the content of the JSON, the structure of the request that our browser made to this HTTP endpoint, and the response that the endpoint returns. Take note of the information below the Request Headers section since we will need it to make our request later.

Clicking the Preview tab will allow us to explore the JSON response with a friendlier user interface.

One thing to note is that the response from the HTTP endpoint gives us richer and structured data about Airbnb properties as well as our search query.

Based on this information, we can also study the pattern and structure of the Airbnb HTTP endpoint.

Finally, we will make an HTTP GET request using Axios to the endpoint. Notice how we also include the headers information that we previously identified. Including these headers will allow Airbnb to see our request as if we are Airbnb.com requesting data from Airbnb’s server.

Next Steps

The execution of this method is simple and easy. However, there is an investment in the effort to do trial and error as well as identifying suitable endpoints. Although, once we find the right pattern and solidify our script, we can use it for any search queries and datasets that we want to obtain!

Beyond the example in this article, we can find other endpoints and retrieve different datasets from the website. Airbnb uses other HTTP endpoints to populate data for different sections of its website. For example, Airbnb uses different endpoints for reviews, which we can also obtain through this technique. Tips: Play around with the Developer Tools with the Network tab open and study the incoming requests. You may find unique HTTP endpoints that you can use.

We can also create a better algorithm for iterating through the endpoints. Instead of the fixed amount of loops as shown on the example, we can create a check that will increment the offset until no data are returned. Finally, play around with the parameter of the endpoint and see what data can be retrieved.

This method is quite versatile and applies to many other popular platforms. YouTube and Instagram are few of many others that utilize this client to server interactions.
